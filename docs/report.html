On 148 rows, an AUC of 0.95 is an accusation, not a result.

The "negatives" here aren't dislikes — they're tracks from my own Made-For-You mixes that I haven't saved. Spotify's recommender already thinks I might like them, which makes this a positive-unlabeled (PU) learning problem and every metric above a lower bound on true discrimination.

How it works

  1. Collect. Liked Songs + four Made-For-You mixes pulled live through the Spotify MCP connector (the audio-features API is gone; titles, artists and durations are what you get).
  2. Enrich. Artist, genre and release year recovered via the iTunes Search API — throttled, cached, fuzzy-matched with confidence scores; unknowns stay NaN rather than guessed.
  3. Featurize. Genre one-hots, era, duration, explicit flag, char-ngram TF-IDF over title+artist. Source-derived features are banned: the label is defined by source membership.
  4. Evaluate honestly. Leave-one-artist-out CV — whole artists held out so the model can't win by memorizing that I like Coldplay. Logistic regression beats gradient boosting here (small-data regularization wins).
  5. Rank. The final model (plus a leave-one-out artist affinity, excluded from evaluation) scores every unsaved track into a top-20 with per-song "why" explanations.

What I'd tell a reviewer

  • PU learning, not classification. Unsaved mix tracks are plausible likes, so 0.615 is a floor, not a ceiling. Getting true negatives needs a blind listening test (planned: rerank vs. Spotify's order, A/B).
  • 148 tracks. Every number carries wide error bars; the point is the methodology, not the third decimal.
  • Selection bias. Candidates come pre-filtered by Spotify's recommender — this model reranks a shortlist, it doesn't search the catalog.
  • Enrichment provenance. Where the network-restricted run couldn't reach iTunes, a hand-curated table (high-confidence only, flagged in the repo) fills artist/genre/year.
